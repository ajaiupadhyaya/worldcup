# Floodlit Prediction UI — Design

**Date:** 2026-06-21
**Status:** Approved (pending user spec review)
**Depends on:** the prediction engine (`model/`) and its committed snapshots in repo-root `data/` (predictions/ratings/calibration). Engine shipped + merged; group labels wired (snapshots non-degenerate).

## Goal

Surface the prediction engine's output in the live Floodlit app, in the FLOODLIT
CHALK design language, leading with **honesty** — uncertainty bands and
out-of-sample calibration are first-class, not hidden. The engine is currently
backend-only (snapshots sit in `data/*.json`, served by no route). This makes it
user-visible.

## Non-Goals (explicitly deferred)

- Homepage "title odds" teaser on `/` (Live).
- Shareable `next/og` prediction card.
- Telestrator-bracket alternate visualization (the survival funnel is the chosen form).
- Consuming the per-slot `bracket[]` data (the funnel uses per-team stage
  probabilities; per-slot data is unused for now).
- Any change to the engine, the cron, or `data/` generation.

## Inputs — snapshot schemas (already produced by `model/`)

All three are committed JSON in repo-root `data/`, regenerated by the 6h cron.

**`data/predictions/latest.json`**
- `generatedAt, modelVersion, seed, simCount, inputsHash` (strings/numbers)
- `thirdsTableComplete` (bool, optional)
- `teams[]`: `{ id (slug), name, qualify, reachR32, reachR16, reachQF, reachSF,
  reachFinal, winCup, mcStdErr: {<stage>: number} }` — sorted by `winCup` desc.
  All stage values are probabilities in [0,1].
- `groups[]`: `{ group: "A".."L", teams: [{ id (slug), finishProbs: {p1,p2,p3,p4} }] }`
- `bracket[]`: `{ slot, teamProbs: [...] }` — **not used in v1**.
- `fixtures[]`: per-match rows — **not used in v1**.

**`data/ratings/latest.json`**
- `teams[]`: `{ id (slug), name, attack, defense, elo, overall, style }` — sorted by `overall` desc.

**`data/predictions/calibration.json`**
- `generatedAt, brier, logloss`
- `reliability[]`: 10 bins of `{ binMid, n, observed, predicted }` (a
  predicted-vs-observed calibration curve; `n` = sample count per bin).

Note: `groups[].teams[]` and `bracket[]` carry only `id` (slug), not display
names. Display names are joined from `predictions.teams[]` / `ratings.teams[]` by slug.

## Architecture

### 1. Data layer — `lib/predictions.ts` (server-only)

- TypeScript types mirroring the three snapshots: `PredictionsSnapshot`,
  `RatingsSnapshot`, `CalibrationSnapshot`, plus row types.
- **Loading: static JSON import at build time** — `import predictions from
  "../data/predictions/latest.json"` (and ratings, calibration). Rationale: the
  cron commits new snapshots → Vercel redeploys → the import is re-bundled. This
  is the existing freshness model. Avoids runtime `fs`, `process.cwd()`, and
  Vercel `outputFileTracing` pitfalls; it is type-safe and fast (data in the JS
  bundle, pages render statically). Requires `resolveJsonModule` (already on for
  Next/TS) and a relative import from `lib/` to repo-root `data/`.
- **Pure, unit-tested transforms** (no React, no I/O):
  - `funnelRows(teams, { perStage })`: for each stage in
    `["reachR16","reachQF","reachSF","reachFinal","winCup"]`, return the top-N
    teams by that stage's probability, each as `{ id, name, prob, stdErr }`.
  - `qualifyByTeam(teams)`: `Map<slug, qualifyProb>` for the standings join.
  - `formatProb(p)`: `0.248 → "25%"` (and a `formatProbPrecise` for small values
    like "0.4%"). Reuse `lib/format.ts` helpers where they exist.
- These transforms are the regression-tested core; everything else is presentation.

### 2. `/predict` hub — `app/predict/page.tsx` (Server Component)

A single vertical chalk scroll, max-width consistent with existing pages, with a
shared `Section` heading treatment (Anton display caps + mono kicker). Sections:

- **WIN THE CUP** — `components/predict/WinCupLeaderboard.tsx`
  All 48 teams ranked by `winCup`. Each row: rank, `Flag` + name, a
  telestrator-cyan probability bar, the percentage (mono, tabular), and a faint
  `± mcStdErr.winCup` tick/label for honesty. Reuses `Flag` + `teamColors`.
- **BRACKET** — `components/predict/SurvivalFunnel.tsx` (signature visual)
  Five stage columns: Round of 16 → Quarter → Semi → Final → Champion. Each
  column shows the top teams by probability of *reaching that stage*, with chalk
  bars (cyan), name + percentage. Built from `funnelRows()`. Honest: it mirrors
  the Monte-Carlo stage probabilities directly. Responsive: columns stack /
  horizontally scroll on narrow screens.
- **RATINGS** — `components/predict/RatingsTable.tsx`
  Table of `ratings.teams[]`: rank, Flag + name, Elo (mono), attack, defense,
  overall. Default sort by overall; column-sortable client-side. `style` is `{}`
  today (deferred) — omit that column until populated.
- **THE MODEL** — `components/predict/CalibrationPanel.tsx`
  Headline Brier + log-loss (mono, with a one-line plain-language gloss), a small
  inline SVG reliability curve (predicted x vs observed y, diagonal = perfect,
  dot radius ∝ `n`), and a methodology line: `generatedAt`, `simCount`, `seed`,
  `modelVersion`, plus a short "Dixon-Coles + Elo, Monte-Carlo over the real 2026
  format; out-of-sample backtest" note. This is the "show your work" footer.

All sections are server-rendered from the static snapshots. Only the sortable
ratings table and any hover affordances are client components.

### 3. `/standings` enhancement

Add a **projected-qualify %** column to `components/StandingsTable.tsx`, sourced
from `qualifyByTeam(predictions.teams)` joined to each live-standings row by team
slug (`lib/format.ts` / existing slug helper; mirror the model's `_slug`:
lowercase, strip apostrophes, spaces→hyphens). Column header e.g. "PROJ. QUALIFY".
Degrades gracefully: if a team has no snapshot entry (slug mismatch / no
snapshot), render "—" for that row and never break the table.

### 4. Navigation

Add `{ href: "/predict", label: "Predict" }` to `LINKS` in `components/SiteNav.tsx`
(after "Groups"). Active-state styling already handled by the component.

## Design language (must match existing)

- Colors via CSS vars (`globals.css`): bars/links = `--home` (telestrator cyan);
  `--accent` (sodium amber) for emphasis/active; `--muted` for axis/secondary;
  never pure white. Cards: `--surface`, hairline `--border`, `--radius-card` (4px).
- Fonts: `--font-display` (Anton, caps) for section headings; `--font-mono`
  (Spline Sans Mono, tabular) for all numbers; `--font-body` (Hanken) for prose.
- Probability bars use the chalk aesthetic (flat fills, hairline tracks). Honor
  `prefers-reduced-motion` (existing pattern); no gratuitous animation.
- Accessibility: visible focus (already global), semantic tables, sufficient
  contrast, `aria-label`s on bars conveying the numeric value.

## Testing & verification

- **vitest** (new dev dependency for the Next app — it currently has none):
  unit tests for the pure transforms in `lib/predictions.ts` — `funnelRows`
  (ranking + top-N + shape), `qualifyByTeam` (slug join, missing-team), and
  `formatProb` (rounding, small values). Tests import a small fixture object, not
  the live snapshot, so they are deterministic.
- **Type + build:** `tsc --noEmit` and `next build` must be clean.
- **Browser:** a Playwright pass over `/predict` (all four sections render with
  real data, no console errors, mobile + desktop) and `/standings` (projected
  column present and populated).

## Risks / edge cases

- **Slug-join mismatches** between live ESPN team names and the model's slugs
  (e.g. "South Korea" vs "Korea Republic"). The standings column must degrade to
  "—" rather than break; note any systematic mismatch for a follow-up alias pass.
- **Static import path** from `lib/` to repo-root `data/` must resolve under
  Turbopack; verify in `next build`, not just dev.
- **Stale/degenerate snapshot**: if a future snapshot regresses to one group, the
  funnel/leaderboard still render (just sparse). No hard dependency on 12 groups.
- **48-row leaderboard length**: keep it scannable (rank numbers, tabular figures,
  maybe a subtle fold after the top contenders) — but show all 48; this is a
  reference surface.
